# 判定ルール集（確信度・カテゴリ・マスク確定）

data-redactor が「何を・どのくらいの確からしさで機密と見なし・どう処理するか」を
1 本にまとめたリファレンス。

確信度（強・中・弱…）の決まり方とマスク確定の手順を、ここで引けるようにする。

- 対象読者：このツールの判定を理解・レビューする人。
- 概要は [README.md](README.md)。設計の経緯は `docs-dev/`（git 管理外）。
- **本書が「現在の挙動」の正**。

---

## 0. 大前提（これだけは外さない）

- **recall 最優先** ＝ マスク漏れ ＝ 情報漏洩。迷うものは伏せるか、レビューに回す。

- **「隠す」だけでデータは消さない**。微弱・除外も候補・データには残る（既定で非表示なだけ）。

- **ラベルは信じない**。種別（人名か社名か）は揺れる前提。芯は「**隠すべき物か否か**」の合議。

- **確定の拠り所は名簿（辞書）だけ**。検出器は誤るので確定にしない。

---

## 1. 全体像（入力 → 判定 → 取り扱い）

```
入力（ファイル / テキスト / kb-mcp）
  → テキスト化・チャンク分割
  → 各チャネルが「票」を出す
       ├ 辞書（dict）      … 名簿一致           → 確定の根拠
       ├ 正規表現（regex） … 連絡先（メール等）  → 強の根拠
       ├ NER 系統          … Sudachi品詞 ∪ GiNZA(electra / ja_ginza)
       └ LLM 系統          … pii-masker（文脈判定）
  → 2系統合議（NER と LLM は対等な2系統）
       ├ ① カテゴリ（種別）を決める
       └ ② 確信度（強 / 中 / 弱 …）を決める
  → ③ 取り扱い（自動マスク / 要レビュー / 非表示）に振り分け
```

要点は **NER と LLM が対等な「2系統」**であること（§3）。

### 辞書（dict）の照合は2段

ユーザーが意識するのは「まるごと／部分」だけ（詳細は `data/mask_dict.sample.yaml`）。

**① 完全一致（既定）** — まるごとの語だけ。より長い識別子の断片では拾わない。

- 拾わない例：`社A` を `社A工場` の中で／`iAS` を `iASMap` の中で／`IF-` を `IF-X` の中で
  （ラテン英数字・連結記号 `-_` が**空白無しで隣接**すると弾く）。
- **空白区切りは語境界**として扱う：`LB SONY EGA` の `SONY` は隣が `LB`/`EGA` でも拾う。
  - トークンの全文オフセットの隙間で「空白で切れている」と判定する
    （NER 経路は空白トークンを落とすため、この判定が必須）。

**② 部分一致（`部分一致: true`）** — 他の語の**中**でも境界沿いに拾い、その部分だけ隠す。

- 区切り入り（`IF-`→`IF-X` の `IF-`）でも camelCase（`iAS`→`iASMap`）でも同じフラグで効く。
- `ECBType` の `CB` は境界外なので不発。
- 実装：`MaskDictionary.match`（完全一致）/ `partial_matches`（部分一致）。

**大小の扱い**

- 既定は大小・全角半角を吸収（`SMASH` = `Smash` = `smash`）。
- **`case_sensitive: true`**（略語向け）にすると大小を区別。
  - `STS` は `STS` のみ拾い、`Sts`（Status の略かも）/`sts` は拾わない。
  - 出現展開（`apply`）も case_sensitive 語は大小区別で広げる（`STS` 確定でも `Sts`/`sts` に波及しない）。
- 除外リスト（allowlist）も同じ `case_sensitive` を持つ。

## 2. カテゴリ（6種）

| カテゴリ | 例 | マスク対象か | 確信度の決まり方 |
|---|---|---|---|
| **人名** | 田中、John Smith | ★特別（対象） | 系統合議（§4） |
| **社名** | オーロラ電子、Nebula | ★特別（対象） | 系統合議 |
| **商標** | 製品名・ブランド | ★特別（対象） | 系統合議 |
| **連絡先** | メール・電話 | 対象（主に regex） | regex 一致＝強 / LLM 単独＝中（§3） |
| **地名** | 横浜市、長崎 | 対象外（弱＝要レビュー） | 必ず弱 |
| **その他** | 識別子・一般語 | 対象外 | 弱 → **微弱**（§9。LLM識別子だけ弱で免除） |

- **特別カテゴリ ＝ 人名・社名・商標**。確信度の「強/中」を投票で決めるのはこの3つ。

- **連絡先の確定的な検出は正規表現（＝強）**（§3。メール等は形が決まっているため）。
  - LLM も連絡先を出すが、単独なら中（regex が主役。取りこぼしを LLM が拾う位置づけ）。

- **地名・その他は票数によらず弱**（誤分類で人物が紛れるので必ず人がレビュー）。

- カテゴリは伏せ字の見た目（`[人物1]`/`[社1]` 等）を決めるだけ。確信度には効かない。

- **人名は辞書登録しない**（人名は無限で列挙できず、名簿として保持もしない）。
  - 辞書は社名/商標が主役、人名は系統（NER∪LLM）で拾う。
  - dict の人名セクションは機構としては存在するが、運用では使わない。

## 3. チャネルと「2系統」

票は `(チャネル, ラベル)` の形。チャネルは独立で、**走ったチャネルだけ**を集約する。

### 確信度を数える2系統（対等）

- **NER 系統** ＝ `sudachi` ∪ `ja_ginza_electra` ∪ `ja_ginza`
  - この3つは**「票」ではなく系統内の入力**。NER は内部で何チャネル一致しても、外に出す意見は **1つ**。
  - （フラットに数えると NER の3チャネルが LLM の1票を圧殺するため、等分にする。）

- **LLM 系統** ＝ `llm`（pii-masker）。常に1系統。

### 系統に数えない「決定的チャネル」

それぞれ固有の確信度床を持つ。

| チャネル | 意味 | 確信度 |
|---|---|---|
| `dict` | 実辞書（名簿）一致 | **確定** |
| `session` | 昇格（他箇所で確認済みの語を再注入） | **強** |
| `regex` | 連絡先（メール等）の正規表現一致 | **強** |

- `session`（昇格）＝ある箇所で強になった clean な語を「確認済み」として全出現へ広げる仕組み。
  - 名簿ではないので**確定ではなく強**に留める。

- `regex` 連絡先は決定的だが誤検出があり得るので**確定でなく強**（除外リストで外せる）。

## 4. 確信度の決まり方 ★この章が核心

確信度は **NER×LLM の2系統合議**で決める。

| 確信度 | 条件 | 扱い |
|---|---|---|
| **確定** | 辞書（名簿）一致 | 自動マスク |
| **強** | **2系統（NER∧LLM）がともに特別カテゴリを検出**（種別が違ってもよい）／ session ／ regex連絡先 | 自動マスク |
| **中** | **1系統のみ**が特別カテゴリを検出（floor：特別は必ず中以上） | 要レビュー |
| **弱** | どの系統も特別を出さない（地名/その他のみ） | 要レビュー（自動マスク外） |
| **微弱** | 中/弱 のうち「コードらしき」表層（§9） | 既定で非表示・自動マスク外 |
| **除外** | allowlist で人が「機密でない」と外した語（§9） | 既定で非表示・自動マスク外 |

### 読み方

- **強** ＝ 2系統（NER∧LLM）がともに「これは隠すべき」と言った ＝ 強い証拠。
  - NER だけが何チャネル一致しても 1系統 ＝ **中**まで。LLM が併走して初めて 2系統 ＝ **強**。
  - ⚠ 前提は「**両系統の誤りが独立**」であること。
    コード識別子では ja_ginza と LLM が相関して誤爆し、ノイズなのに 2系統一致＝強に化けることがある（§9）。
    「強＝常に信頼」ではない。

- **中** ＝ 片方の系統だけ ＝ **レビュー必須**。floor があるので特別カテゴリは弱に落ちない。

- **LLM 単独は構造上 最大「中」**（1モデルなので第2の意見が無い）。欠陥ではなく当然の帰結。

### 当てはめ例

| 候補 | NER系統 / LLM系統 | カテゴリ | 確信度 |
|---|---|---|---|
| Sony | 社名 / 社名 | 社名 | 強（2系統） |
| 種別割れ | 社名 / 人名 | 人名（重い方） | 強（2系統・種別違いでも可） |
| 佐伯 | 人名（NER単独）/ 沈黙 | 人名 | 中（1系統） |
| 小浜 | 地名 / 人名 | 人名（特別が地名に勝つ） | 中（人名は LLM 1系統） |
| 長崎 | 地名 / 地名 or 沈黙 | 地名 | 弱（特別なし） |

### 確信度は「出現ごと」／自動マスクの選択は「実体単位」

- **確信度は出現（span）ごと**に決める（上の合議は各出現で走る）。
  - 同じ表層でも、NER が当たった出現だけ 2系統＝強、他は LLM 単独＝中、と割れ得る。

- **自動マスクの初期選択は実体単位**：ある表層に 確定/強 の出現が 1 つでもあれば、その表層の**全出現**を選ぶ
  （`_auto_mask_spans`。CLI の `apply(expand=True)` と同じ）。
  - 例：`C社ファイル` が 4 出現、うち 1 出現だけ NER＋LLM で強・残り 3 は LLM 単独で中。
    → 実体は社名・強 → **4 出現すべてを自動マスク**（1/4 しか隠れず 3 箇所漏れる、を防ぐ＝recall）。

- **同形異義語は壊れない**：`小浜`（`小浜市`=地名 / `小浜さん`=人名）は、どの出現も 2系統一致に
  ならず**中止まり**なので、この自動選択（強のみ）に引っかからない。出現ごとの区別は保たれる。
  - 一方の出現が真に強な同形異義語（例 人名 `小浜さん` が NER＋LLM で強）は、実体単位ゆえ他出現も
    初期選択に含まれるが、**出現ごとビューで個別に外せる**（§7）。

## 5. カテゴリ（種別）の決まり方

優先度：**人名 > 社名 > 商標 > 連絡先 > 地名 > その他**（`_CAT_PRIORITY`）。

### 5-0. 入力ラベル → カテゴリ

各チャネルの生ラベルを 6 カテゴリへ写す
（`_NER_LABEL_CATEGORY` / `_ENE_TO_CATEGORY` / `_sudachi_category`）。

**ここで拾わないラベルは候補にすら上がらない**（＝票にならない）。

#### NER（GiNZA electra / ja_ginza。関根の拡張固有表現ラベル）

| カテゴリ | 採用するラベル | 備考 |
|---|---|---|
| 人名 | `PERSON` | `N_PERSON`（人数）は数値表現なので含めない |
| **社名** | `COMPANY` / `COMPANY_GROUP` | **この2つだけ**。組織系（CORPORATION_OTHER 含む）は含めない |
| 地名 | `CITY` / `PROVINCE` / `COUNTRY` | 粗い都市/県/国のみ。下記は含めない |

**社名として採らない組織系ラベル**

- `CORPORATION_OTHER`（企業その他）/ `SHOW_ORGANIZATION`（興行団体）/ `SPORTS_ORGANIZATION_OTHER` /
  `POLITICAL_ORGANIZATION_OTHER` / `POLITICAL_PARTY` / `GOVERNMENT` / `INTERNATIONAL_ORGANIZATION`
- 業務文書で一般語の誤爆（ゴミ）になりやすいため（`CORPORATION_OTHER` も粒度が粗く社名として怪しい）。
- 実在の組織は **LLM(Company)＋辞書**で拾う。

**地名として採らない `_OTHER` 系**

- `GPE_OTHER` / `GEOLOGICAL_REGION_OTHER` / `LOCATION_OTHER` / `DOMESTIC_REGION_OTHER` /
  `CONTINENTAL_REGION_OTHER` / `FACILITY_OTHER`
- 粒度が粗く誤爆しやすい。

**地名として採らない詳細ラベル**

- `FACILITY_PART`（「3階」「西口」等の施設部位＝ほぼ純ノイズ）
- `STATION` / `AIRPORT`（駅/空港）
- `ADDRESS` / `POSTAL_ADDRESS`（住所/郵便番号）
- 理由：候補一覧のノイズになるわりにレビュー価値が低い。
  **地名は元々 弱＝自動マスク外なので、外してもマスク漏れは起きない**
  （人名/社名/商標が言えばそちらで拾う）＝効果はノイズ削減のみ。
  - どうしてもマスクしたい地名は辞書登録で確定へ上げる。
  - LLM 側の `Address`/`Facility` は地名として据え置く
    （LLM 経路は未登録 type が「その他」に落ちてノイズが移るだけで消えないため。NER 経路だけ絞る）。

**その他 NER で拾わないもの**

- 連絡先は NER から取らない（`@`・数字に過剰発火するため regex に委ねる）。
- `Product_Other` 系はノイズ過多で除外。
- 国籍・民族・一族（`Nationality`/`Ethnic_Group_Other`/`Family`）は会社でないので社名にしない。

> NER 単独の社名/人名票は元々**中**（自動マスク外）。上の絞り込みの主効果は**レビュー一覧のノイズ削減**
> （recall への実害は小。地名はそもそも弱＝レビュー）。

#### LLM（pii-masker の ENE type）

| ENE type | カテゴリ |
|---|---|
| `Person` | 人名 |
| `Company` / `Department` | 社名 |
| `Province` / `City` / `Country` / `Address` / `Facility` | 地名 |
| `Email` / `Phone_Number` | 連絡先 |
| `Trademark` | 商標 |
| `Employee_ID` / `Account` / `IP_Address` | その他（識別子） |

#### Sudachi 品詞

| 品詞 | カテゴリ |
|---|---|
| `固有名詞-人名` | 人名 |
| `固有名詞-地名` | 地名 |
| `固有名詞-一般` | その他（未知の英字を社名に水増ししない） |

#### 決定的チャネル

- `dict` ＝ 辞書のセクション（社名/商標/人名。ただし**人名は運用では登録しない**・§2）
- `session` ＝ 昇格元のカテゴリ
- `regex` ＝ 連絡先（§3）

### 5-1. 合議でカテゴリを決める

**1. 辞書一致があれば、そのカテゴリで確定**（最優先・名簿が砦）。

- **足場（確定＝辞書 / 昇格＝session）スパンと少しでも重なる非足場候補は、候補として出さない**。
  - 内包（`SONY` を覆う `SONYビル`）
  - またぎ（`SONY・Nikon・Canon` を覆う粗い NER）
  - **部分的に重なるだけ**の候補も落とす
    - 例：NER が `情報.D` を1エンティティ化し、辞書の部分一致 `D-Cap`[3,8) と [3,4) だけ重なる場合、
      **確定 `D-Cap` だけを候補に出し、`情報.D` は出さない**。
    - 狙い：確定で消える箇所に重なる別スパンを二重に出さない（表示・選択の混乱と確信度漏れを断つ）。
  - この抑制は「重なるから」であって特定語の狙い撃ちではない。
    確定アンカーが無い箇所なら同じ `情報.D` は NER 単独の中候補として通常どおり登場する
    （一般語の誤検出は allowlist / NER の precision 課題）。

- 決定的票（辞書/昇格/連絡先 regex）は、その一致が**当の候補スパンに収まる**ときだけ確信度に効く。
  - NER/LLM の系統票は重なるスパンから集めて確信度に効かせる（橋渡し）。

**2. 辞書がなければ各系統を 1カテゴリへ畳む**（系統内合議）。

- 系統内でも「特別が地名/その他に勝つ・重い特別が勝つ」＝優先度最上位を採る。

**3. 系統間でも同じ原理**：いずれかの系統が特別を出したらそれを採用（地名/その他に埋もれさせない）。
両系統が別の特別なら**重い方**。

### 5-2. Stage 1：NER の特別票を surface で弱める（コードノイズ対策）

ja_ginza 等が英字/コードを社名・人名へ誤爆するので、NER の特別票を surface で「その他」に落とす。

狙い：**コードノイズが LLM と組んで「強」になる経路を断つ**（`_system_category`）。

判定は2軸で弾く（OR）。カテゴリ非対称：

- `_looks_like_code` ＝ 記号/数字の「コード性」（カテゴリ非依存）
- `_looks_like_nonperson_latin` ＝ ラテン文字の「人名らしさの否定」（**人名専用**）

**人名を弱める条件**：`_looks_like_code` または「ラテン文字主体で Titlecase 語の並びでない」→ その他。

実在のラテン人名は Titlecase（`John`/`Smith`/`Anne-Marie`）。それ以外は識別子/術語の誤爆として弾く：

- 全小文字（`theta`/`list`）
- camelCase・PascalCase の途中コブ（`EndTime`/`MarkID`/`dElem`/`OutputItegrity`）
- 全大文字（`FIARSL`）／2文字以上の全大文字語を含む複合（`Wafer ID` の `ID`）
- 記号・ブラケット混じり（`theta・「`）

守るもの：

- `John`/`Smith`/`Anne-Marie` は Titlecase ＝人名のまま残す（純ASCIIを一律降格しない）。
- かな/漢字を含む人名（`佐伯`）は信頼して残す。
- ⚠ `McDonald`/`DeWitt` 等の途中大文字の外国姓は巻き込む（LLM 2系統＝強・辞書で救済される前提）。
- `サブエラーコード` 等のカタカナ汎用語は表層で固有名と区別できず対象外＝辞書/除外リストの仕事。

**社名を弱める条件**：`_looks_like_code`（記号・英数字・数字のみ・1文字）**のみ** → その他。

- `_looks_like_nonperson_latin` は社名に当てない
  （`IBM`/`SAP`/`Apple`/`eBay`/`PayPal` 等＝Titlecase でない正当な社名を守る）。

**LLM 票は弱めない**（文脈判定を尊重）。

> 結果：NER 単独のラテン人名ノイズ（`EndTime`/`theta・「` 等）は「その他・弱」に収まり、一覧の
> 「人名」から外れる。本物の Titlecase 人名（`John`）は残る。

## 6. 取り扱い（マスク方針）

①カテゴリ ②確信度 を受けて、どう処理するかを決める。

| 確信度 | 取り扱い |
|---|---|
| **確定・強** | 自動マスク（既定でチェック ON。`AUTO_MASK_CONFIDENCE = ("確定","強")`） |
| **中・弱** | 要レビュー（人が確認して付け外し。自動マスク外） |
| **微弱・除外** | 既定で非表示・自動マスク外（データには残る） |

- **中・弱を自動マスクせずレビューに回す**のは recall 最優先の帰結
  （迷うものは人が確認して取りこぼしを拾う）。
- **弱（地名/その他）は自動マスクしない**（utility 保持）。

## 7. 実体ごと と 出現ごと

UI には「実体ごと」「出現ごと」の2ビューがある。

- **マスクの最小単位は出現**（共有の選択集合 `mask_sel`）。
- 両ビューはこの 1 つの集合を読み書きするので、切替で選択は消えない。

読み方の注意：

- **実体ごと**の行に出る確信度は、その語の**全出現のうち最良（代表値）**。

- **自動マスクの初期選択は実体単位**（§4）：実体に 確定/強 の出現が 1 つでもあれば、その表層の**全出現**を初期選択する。
  - 例：`C社ファイル` が 4 出現、うち 1 が強・3 が中 → 実体は強 → **4 出現すべて自動選択**（「全 4/4」）。

- **「⚠ 一部 n/全」** は、**人が出現ごとビューで一部だけ選択/解除した**ときに出る（自動マスクの取りこぼしではない）。
  - 同形異義語（`小浜市` は隠す・`小浜さん` は残す 等）を出現単位で作り分けたケース。

- 実体チェックは**全出現が揃ったときだけ ON**（一部選択なら OFF ＝ ⚠一部 表示）。

## 8. マスク確定の方法（手順）

1. 文書を読み込み、各タブ（NER 検出／LLM 検出）で検出を走らせる。

2. **マージ&確信度**タブで票を集約し、確信度づけ（§4）する。

3. 既定で **確定・強**にチェックが入る（自動マスク候補）。中・弱はレビュー候補として一覧表示。

4. レビューして、チェックを付け外し（実体ごと/出現ごと）→ **[✅ マスクを反映]**。
   - 「機密でない」と判断した語は **[🚫 選択を除外リストへ]** で allowlist に送る（§9・除外）。
   - 「常に伏せたい社名・商標・人名」は **[📒 選択を辞書へ登録]** で辞書に送る（その語のカテゴリで登録）。
     GiNZA/LLM はキャッシュのままその場で再マージし、登録語は**確定**（=自動マスク）として即反映される。
     カテゴリが 地名/連絡先/その他 の語は辞書に登録できない（辞書は社名/商標/人名のみ）。

5. 反映後の**マスク済みテキスト**と、**復元用の対応表（placeholder↔原語）**を得る。
   - プレースホルダは**表記（surface）ごと**に振る。表記が違えば別（eXmotion→[社1] /
     エクスモーション→[社2]）、同じ表記の繰り返しは同一。これで復元が元の表記へ正確に戻る。
   - 例外：辞書で**固定の置換語（mask:）**を指定した実体は、表記ゆれもその 1 つの語に寄せる
     （復元はその実体の代表語へ。表記ゆれは保存されない）。

> レビューの付け外し（手動の差分）は **`mask_draft` として content_hash 単位で永続化**。
> 再起動・再解析・ビュー切替でも手動選択は消えない。

## 9. 微弱・除外（候補から消さず、既定で隠す2つ）

### 微弱

「コードらしき」表層を中/弱から降格（社内コード/変数名/列挙子の誤ラベル対策）。

**`_looks_like_code` が次のいずれかに該当したら微弱へ**（判定前に先頭/末尾の引用符類を剥がす）：

- **記号マーカーを含む** — `_` `::` `@` `~` `[` `]` `{` `}` `=` `!` `<` `>` `;` `|` `\`
  - 例：`Em_OffOn::idOff` / `ピッチ@` / `~C02` / `37D]==0`
  - `\` はファイルパス由来（`C:\Group\ABC-Z\...` を NER が `Group\ABC-Z` と塊化）。実在名に `\` は出ない。

- **ASCII のみ＋数字を含む** — `16D` / `1L` / `37D`（実在の人名・社名はまず数字を含まない）

- **数字・記号のみ**（名前に使う字を含まない）— `7-410`

- **漢字以外の1文字**

- **CJK区切り・括弧が ASCII英字に隣接** — `AP・` / `theta・「` / `会社「A」`（スパン境界の人工物）
  - 隣接に限る：`C型補正値リミット・チェック結果`（区切りが仮名間）や
    `ジョン・スミス`/`ソニー・ミュージック`（純カナの中黒名）は**守る**。

- **先頭・末尾に垂れたハイフン類・中黒・ピリオド** — `AP-` / `R-` / `-C02` / `田中・` / `情報.`
  - `情報.` は文末/フィールド区切りの `.` を NER が固有表現に巻き込んだもの。
  - 内部の `Coca-Cola` / `Anne-Marie` / `Booking.com`、長音 `ー` は**守る**（端だけ見る）。
  - 対象は半角 `.` と全角 `．`（`。` は既にスパン分割で除去済み）。

- **不均衡な丸括弧**（open ≠ close）— `LSMonito(` / `)Monitor` / `東芝（`
  - NER スパンが括弧を巻き込んだ人工物。
  - ブラケット `[ ] { }` は常にコード。丸括弧 `( )`（ASCII）・`（ ）`（全角）は `Sony (Japan)`/`会社（日本）`
    のような正当な社名に出るので、**数の不均衡だけ**を artifact とみなす（均衡した括弧つき社名は守る）。

- 引用符の扱い：判定前に**先頭/末尾の引用符類を剥がす**（全角/スマート引用符 `＂` `“` `”` は ASCII 判定を素通りするため）。
  - 剥がした中身で判定 ＝ `"O1234.01` はコード判定に載り、`"ソニー"` は実名なので落ちない。

**さらに微弱へ落とすもの**

- **全大文字ASCII（区切り込み。`WEXPC-YCD`）**
  - ただし **社名・商標は除外**（`IBM`/`SAP` の略語社名・`MCSX` 等の全大文字な商標/型番を守る）。
  - 人名の全大文字は Stage 1 でその他へ落ちる。

- **「その他」カテゴリ全般（＝弱）**（隠すべきか不明な低価値）
  - Sudachi `固有名詞-一般` の「ただの固有名詞」（`装置A` 等）や、
    Stage 1 で特別→その他へ降格されたノイズ（`EndTime` 等）はここで既定非表示に。
  - **地名（弱）は「その他」でないので据え置き**（要レビュー）。

**適用範囲**

- **チャネルは問わない**（NER 専用ではない）。中/弱 かつ コードらしき なら NER・LLM どちらの検出でも降格。
  - ＝ 識別子以外の LLM 検出（人名/社名/商標）が中・コードらしきなら、LLM 由来でも微弱に落ちる。

- **唯一の免除 ＝ LLM が識別子（Employee_ID / Account / IP_Address）と判定したもの**。
  - 中/弱でも降格せず弱で残す（`7-410` 型の社員番号が消えると recall 漏れ＝致命的なため）。

- **確定/強は降格しない**（チャネル問わず）。
  - コードらしき語が強に化けないよう Stage 1（§5-2）で NER 票を先に弱めるので、
    記号/英数字コードはそもそも強に到達せず中→微弱に落ちる。

### 除外

allowlist で人が「機密でない」と外した語（最弱）。

- 文脈非依存・グローバル。既定は**完全一致**（正規化 surface）。既定で非表示・自動マスク外。

- **`部分一致: true`**（辞書と照合ロジックを共有 ＝ `dictionary.contains_partial`）
  - 登録語を**境界で内包**する候補を**丸ごと除外**する。
    - 例：`FB`→`GetFBData`/`FBData`/`FB_DATA`、`補正`→`用補正値`、`IF-`→`IF-X`
  - 辞書 `部分一致` と同じアトム分解（ラテン subword＋区切り記号＋CJK は 1 トークン内で束ねる）。
    区切りまたぎ（`IF-`）も camelCase も CJK 形態素も同じに扱う。
  - 単位の**途中は拾わない**（`FBI` の `FB` / `補正` の一部 `補` は不一致＝境界照合）。
  - 形態素境界を厳密に使うには `apply_allowlist` に候補を覆う `tokens` を渡す。
  - **辞書との違いは効き方だけ**：辞書 `部分一致` は一致部分だけマスク、除外は候補単位＝スパンを割らず丸ごと落とす。

- **守るのは辞書（dict）票のみ**（`_has_dict_vote`）＝ **確定 ＞ 除外**。
  - 辞書語は人が意図して「必ずマスク」と登録した名簿なので、除外リストで上書きしない。
  - **連絡先 regex（強）や NER/LLM 検出は検出由来なので除外できる**
    （誤検出メール等を人が明示的に外せる。表層単位なので本物のメールには波及しない）。
  - 除外 `部分一致` と辞書 `部分一致` が同じ複合語に当たっても辞書が勝つ
    （例：辞書 `Smash`部分一致 × 除外 `Type`部分一致 の `SmashMarkType` → `Smash` は伏せ・Type 除外は効かない）。

- ⚠ **候補丸ごと除外の recall 注意**：部分一致 除外はスパンを割らず候補全体を外す。
  - 実在社名が NER で 1 スパンに塊化し、その中に部分一致キーが当たると漏れる
    （`ABC` 部分一致 → 実在社名 `Group\ABC-Z` を丸ごと除外）。
  - **社名/商標は辞書で守るのが基本**（辞書アンカーはクラスタ分割で先に切り出され確定になる＝除外が触れない）。
  - 部分一致は「機密名に出てこないと確信できる語」限定にする。

- **完全一致 vs 部分一致 の使い分け**
  - 部分一致はキーを含む**語族すべて**を外す（`オプション` 部分一致 → `ティーチングオプション` も `○○オプション` も全部）。
  - **汎用語**（`オプション`/`ステップ`/`本パッケージ` 等）は機密名に紛れ込む恐れがあるので、
    その特定の語だけ外したいなら**完全一致**で登録する。
  - 「この語は機密名に絶対出ない・語族ごと外してよい」と言い切れるときだけ部分一致にする。
